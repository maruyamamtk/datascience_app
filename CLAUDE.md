## 0. 最優先の原則（このプロジェクトの背骨）

1. **深掘りは Level 制（例外なし）** — 全トピックを SPEC §4.1.5 の Level 0〜6 構造で作る。
   浅い解説で「完成」としない。最低でも L0〜L2 を満たし、Level 枠は必ず用意する。
   基準: https://zenn.dev/green_tea/articles/0bce4d47ab0674
2. **可視化は『アルゴリズム図鑑』スタイル** — コマ送り + 色ハイライト + 近傍コールアウト。
   基準スクショ: `/Users/michika_maruyama/Desktop/kindle_coding/screenshots_books/アルゴリズム図鑑`
3. **操作→数式の強連動** — グラフだけでなく数式中の該当項の数値・ハイライトも実時間連動させる。
4. **数式を誤魔化さない、ただし高校数学に橋渡しする** — 導出は全ステップ折りたたみで提示。
5. **コンテンツは Bookmemo を一次情報に** — 該当章の `詳細_*.md` を必ず参照（SPEC §11）。憶測で書かない。

## 1. コンテンツ実装時の注意

- 新トピックは **Level 枠のテンプレート**から作る。Level を省略した薄いページを作らない。
- 各 Level に「理論 → 導出/証明 → 図解/操作 → 応用 → （コード）→ 数値実験 → まとめ」を可能な限り揃える。
- 専門用語・定義・定理は**ノード化**し相互リンクで結ぶ（証明の後追いを保証）。リンク切れを残さない。
- トピック冒頭に**対象読者ガイド**（どの Level からどう読むか）を置く。
- 前提関係を明示（例: 因果推論 L4 は回帰 L1 が前提）し、前提トピックへリンクする。
- 数式は KaTeX。**各項に id を付与**し、操作イベントから参照・ハイライトできる構造にする。
- **網羅性は SPEC §3 のチェックリストで担保**。各トピック実装時に該当タグ（【準1】【Ex】【発展】【出典:◯◯】）を消し込み、欠落ゼロを保証する。公式範囲表（準1級/DSエキスパート/DS発展, SPEC §11）と Bookmemo の手法を**取りこぼさない**。とりわけ **準1級【準1】は必達**。

## 2. 可視化部品の実装規約

- 共通フォーマット = **コマ送り（自動再生＋手動前後送り）／色ハイライト／近傍コールアウト**。
  新規可視化はこの共通部品を使い回し、トピックごとに自作して様式がバラけるのを避ける。
- **3層に疎結合分離**: 「コントロール（操作系）」「計算（純関数）」「描画（グラフ・数式）」。
  計算層は副作用のない純関数とし、単体テスト可能にする。
- 操作値・派生値は**単一の state ストア**に集約（single source of truth）。グラフ・数式・数値表示は同じ state を購読。
- まず**小さな具体例（要素数個）**で原理を見せてから一般化する（アルゴリズム図鑑の運び）。
- 操作は **60fps 目標**。重い計算は再計算を間引く/メモ化。MVP 範囲の統計計算は JS で完結させる。
- **描画層を Web に閉じて疎結合に保つ**（将来のネイティブ化で再利用するため）。

## 3. 技術・アーキテクチャ上の注意

- スタック（暫定確定）: Next.js(App Router)+TS+Tailwind / MDX / KaTeX / D3 / Supabase / PWA。
  逸脱する場合は SPEC を更新し理由を残す。
- **スマホ・PC 両対応を常に確認**（レスポンシブ、タッチ操作対応）。PWA のオフライン閲覧を壊さない。
- 重い計算（MCMC, LightGBM 学習等）は MVP では扱わない。将来 Pyodide/サーバ実行を検討（先走らない）。
- 認証・進捗・正誤履歴は Supabase（RLS 必須）。ユーザーデータの取り扱いに注意。

## 4. 進め方（グローバル指示の具体化）

- **非自明な変更はプランニング**してから着手。3ステップ以上 or 設計判断を伴うものが対象。
- **検証してから完了**。動作の証明（実機/ブラウザ確認・スクショ・テスト）なしに「完了」としない。
- 修正を受けたら `tasks/lessons.md` に教訓を追記し、同じミスを繰り返さない。
- 重要な決定は SPEC.md / tasks/ のマークダウンに記録する。
- **日本語で応対**する。

## 5. やってはいけないこと

- Level を省いた / 図解のない / 出典を無視した「薄いトピック」を量産すること。
- 可視化の様式をトピックごとにバラバラに自作すること（共通部品を使う）。
- 数式の結論だけ書いて導出を省くこと（高校数学への橋渡しを欠くこと）。
- スコープ外（ファイナンス/キャリア論等の Bookmemo）を勝手に取り込むこと。
- MVP に重い計算やネイティブ配信を前倒しで盛り込み、検証を遅らせること。
